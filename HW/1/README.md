# Домашнее задание 01: Система бронирования отелей

## Описание задания

**Вариант 13** - Система бронирования отелей (https://www.booking.com/)

### Данные системы

Приложение должно содержать следующие данные:
- Пользователь
- Отель
- Бронирование

### API требования

Система должна реализовывать следующие API:
- Создание нового пользователя
- Поиск пользователя по логину
- Поиск пользователя по маске имя и фамилии
- Создание отеля
- Получение списка отелей
- Поиск отелей по городу
- Создание бронирования
- Получение бронирований пользователя
- Отмена бронирования

## Архитектурное решение

### Роли пользователей

1. **Гость (Guest)** - обычный пользователь, который:
   - Ищет отели по различным критериям (город, даты, цена)
   - Просматривает информацию об отелях
   - Создает бронирования
   - Управляет своими бронированиями
   - Получает уведомления о статусе бронирований

2. **Менеджер отеля (Hotel Manager)** - представитель отеля, который:
   - Добавляет и редактирует информацию об отелях
   - Управляет номерами и их доступностью
   - Просматривает бронирования своего отеля
   - Обрабатывает запросы гостей

3. **Администратор системы** - управляет системой:
   - Модерирует контент
   - Управляет пользователями
   - Настраивает систему

### Внешние системы

1. **Платежная система (Payment Gateway)** - Stripe, PayPal
   - Обработка платежей за бронирования
   - Возвраты при отмене
   - Безопасное хранение платежной информации

2. **Email сервис** - SendGrid, AWS SES
   - Отправка подтверждений бронирования
   - Напоминания о предстоящих заездах
   - Уведомления об отмене

3. **SMS сервис** - Twilio
   - SMS-подтверждения бронирований
   - Срочные уведомления

4. **Сервис карт** - Google Maps API
   - Отображение расположения отелей
   - Расчет расстояний до достопримечательностей
   - Геокодирование адресов

### Контейнеры системы

#### Фронтенд слой

1. **Web Application** (React + TypeScript)
   - Single Page Application для работы через браузер
   - Адаптивный дизайн для различных устройств
   - Поиск отелей с фильтрами
   - Интерфейс бронирования

2. **Mobile Application** (React Native)
   - Нативное мобильное приложение
   - Push-уведомления
   - Те же возможности что и веб-приложение

#### Бэкенд слой

3. **API Gateway** (Kong / Nginx)
   - Единая точка входа для всех API запросов
   - Аутентификация и авторизация (JWT)
   - Rate limiting
   - Маршрутизация запросов к микросервисам
   - SSL/TLS терминация

4. **User Service** (Python + FastAPI)
   - Регистрация и аутентификация пользователей
   - Управление профилями
   - Поиск пользователей по различным критериям
   - Управление ролями и правами доступа

5. **Hotel Service** (Python + FastAPI)
   - Управление информацией об отелях
   - CRUD операции для отелей и номеров
   - Поиск отелей по городу, названию, фильтрам
   - Управление доступностью номеров
   - Интеграция с сервисом карт для геолокации

6. **Booking Service** (Python + FastAPI)
   - Создание бронирований
   - Проверка доступности номеров
   - Получение списка бронирований пользователя
   - Отмена бронирований
   - Управление статусами бронирований
   - Координация с другими сервисами

7. **Payment Service** (Python + FastAPI)
   - Интеграция с платежными системами
   - Обработка платежей
   - Обработка возвратов
   - Хранение информации о транзакциях

8. **Notification Service** (Python + FastAPI)
   - Отправка email уведомлений
   - Отправка SMS уведомлений
   - Подписка на события через Message Queue
   - Шаблоны уведомлений

#### Слой данных

9. **User Database** (PostgreSQL)
   - Данные пользователей
   - Учетные записи и профили
   - История аутентификации

10. **Hotel Database** (PostgreSQL)
    - Информация об отелях
    - Номера и их характеристики
    - Доступность номеров

11. **Booking Database** (PostgreSQL)
    - Бронирования
    - История изменений статусов
    - Связь с пользователями и отелями

12. **Cache** (Redis)
    - Кэширование результатов поиска отелей
    - Сессии пользователей
    - Часто запрашиваемые данные
    - Уменьшение нагрузки на базы данных

13. **Message Queue** (RabbitMQ)
    - Асинхронная обработка событий
    - Публикация событий о создании/отмене бронирований
    - Обеспечение надежной доставки уведомлений

### Основные сценарии взаимодействия

#### 1. Создание бронирования (реализовано в Dynamic диаграмме)
1. Гость выбирает отель и номер в веб-приложении
2. Заполняет данные бронирования (даты, количество гостей)
3. Запрос отправляется через API Gateway
4. Booking Service проверяет доступность номера через Hotel Service
5. Получает данные пользователя через User Service
6. Инициирует платеж через Payment Service
7. Payment Service обращается к платежной системе
8. После успешного платежа сохраняет бронирование в базу данных
9. Публикует событие "BookingCreated" в Message Queue
10. Notification Service получает событие и отправляет уведомления
11. Пользователь получает подтверждение

#### 2. Поиск отелей
1. Гость вводит критерии поиска (город, даты, количество гостей)
2. Запрос идет через API Gateway к Hotel Service
3. Hotel Service проверяет кэш Redis
4. Если данных нет в кэше, запрашивает из Hotel Database
5. Фильтрует отели по доступности на выбранные даты
6. Обогащает данные геолокацией через Maps Service
7. Сохраняет результат в кэш
8. Возвращает список отелей клиенту

#### 3. Отмена бронирования
1. Гость запрашивает отмену через приложение
2. Booking Service проверяет возможность отмены (политика отмены)
3. Обновляет статус бронирования
4. Инициирует возврат через Payment Service
5. Публикует событие "BookingCancelled"
6. Notification Service отправляет уведомление об отмене
7. Hotel Service обновляет доступность номера

### Технологии

**Frontend:**
- React + TypeScript (Web)
- React Native (Mobile)
- HTTPS/REST для API запросов

**Backend:**
- Python + FastAPI (все микросервисы)
- Kong/Nginx (API Gateway)
- JWT для аутентификации

**Data Layer:**
- PostgreSQL (реляционные данные)
- Redis (кэш и сессии)
- RabbitMQ (очередь сообщений)

**Protocols:**
- HTTPS/REST - взаимодействие между фронтендом и бэкендом
- JDBC/SQL - взаимодействие с базами данных
- AMQP - работа с очередью сообщений
- Redis Protocol - работа с кэшем
- SMTP - отправка email
- HTTPS/REST - интеграция с внешними системами

### Преимущества архитектуры

1. **Масштабируемость** - микросервисная архитектура позволяет масштабировать каждый сервис независимо
2. **Отказоустойчивость** - сбой одного сервиса не влияет на работу всей системы
3. **Производительность** - использование кэша и асинхронной обработки
4. **Гибкость** - легко добавлять новые функции и сервисы
5. **Разделение ответственности** - каждый сервис отвечает за свою область

## Диаграммы

### System Context (C1)
Показывает систему в контексте пользователей и внешних систем

### Container (C2)
Показывает внутреннюю структуру системы: контейнеры и их взаимодействие

### Dynamic - Create Booking
Показывает детальную последовательность создания бронирования с взаимодействием всех компонентов системы
